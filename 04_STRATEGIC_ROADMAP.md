# Strategic Roadmap — Lesson Loom

## Purpose

This roadmap keeps the build focused on contest success while preserving future PrairieSignal value.

Lesson Loom is a **Labs prototype** that supports PrairieSignal Research and Edge. It should not become a full education platform during the contest sprint.

## Strategic placement inside PrairieSignal

- **Labs:** prototype and public demo.
- **Research:** demonstrates AI-era learning workflow design and process evidence.
- **Edge:** future workshop/resource for educators and instructional teams.
- **Studio:** future resource portal or lesson-interface sprint for education partners.

## Contest objective

Submit a high-fidelity interactive prototype that shows the core transformation:

`static lesson plan → interactive student app + teacher console + export pack`

## Highest-probability success path

Build one outstanding lesson transformation rather than many shallow examples.

The selected demo is **Fraction Garden** because it is:

- visually clear;
- interactive;
- low controversy;
- easy to understand;
- easy to differentiate;
- easy to assess;
- suitable for desktop and mobile UI.

## Day-by-day build sprint

Outcomes:

- Core sections scaffolded.
- Design tokens implemented.

Outcomes:

- Teaching signal cards activate after weave.

## Post-contest roadmap

### Phase 1 — Public demo and PrairieSignal proof asset

- Publish as a PrairieSignal Labs demo.
- Add case study page.
- Label accurately: Internal Prototype / Public Demo / Labs Pilot Candidate.
- Add inquiry CTA for education partners.

### Phase 2 — Workshop artifact

Turn Lesson Loom into an Edge workshop demo:

- “From Static Lesson to Interactive Learning Interface.”
- Audience: teachers, teaching centres, instructional designers.
- Artifact: prompt pack, review checklist, sample lesson interface.

### Phase 3 — PrairieClassroomOS discovery

Use the demo to test interest in a broader product thesis:

- lesson interface generation;
- assessment checkpoints;
- process evidence;
- printable fallback;
- classroom workflow review.

Do not build full platform until pilot users validate the pain.

### Phase 4 — Pilot prototype

Only after user interviews:

- add real lesson plan ingestion;
- add export to Google Slides/Docs/PDF;
- add safe curriculum alignment workflow;
- add teacher review gates;
- avoid student accounts until privacy/compliance model is mature.

## Kill / pause criteria

Pause product expansion if:

- teachers like the demo but do not want to use it;
- it requires too much manual curriculum validation;
- it becomes a generic lesson-plan generator;
- privacy or district procurement burden dominates;
- it distracts from Studio/Edge revenue with no partner signal.

## Strategic note

The contest build should create proof and authority, not a maintenance burden.

Treat this as:

- a polished prototype;
- a public proof artifact;
- a workshop wedge;
- a future Labs option;
- not a committed SaaS company.
